# ADR-005: 熔断与安全策略

## 状态

已采纳 · 2025-08-15

## 背景

yuleASR-Configurator 的条件引擎和依赖图系统处理的表达式和规则来自两个方面：

1. **从 ARXML 自动提取的 Schema** — 可能包含递归或循环的 `visibleWhen` 条件。
2. **用户自定义的传播规则** — 可能包含人为错误导致的循环依赖或指数级深度嵌套。

如果不对递归深度加以限制，恶意或错误的输入可能导致：

- **栈溢出**：Node.js/V8 的默认调用栈大约 1M~2M，深度递归几十万层即可触发 RangeError。
- **无限循环**：`visibleWhen` 表达式 `A.visibleWhen = "B.enabled"` +
  `B.visibleWhen = "A.enabled"` 在 UI 渲染中形成无限递归。
- **死循环检测**：`ConstraintPropagator` 中的传播规则出现 A → B → C →
  A 的循环依赖，导致传播在任何变更上都无限循环。

## 决策

采用三层熔断机制（Circuit Breaker），在不同层级施加不同严格度的保护阈值：

### Layer 1: 条件表达式求值深度限制（MAX_DEPTH = 50）

**位置**：`conditions/evaluator.ts:ConditionEvaluator`

```typescript
static readonly MAX_DEPTH = 50;
```

- **触发条件**：`evaluateNode()` 递归深度超过 50。
- **行为**：抛出带有明确原因的 Error（"MAX_DEPTH exceeded in condition
  evaluation. Possible malformed expression."）。
- **消费处理**：`ValidationPipeline` 捕获该异常，包装为 `CONDITION_ERROR` 级别的
  `ValidationError`，不冒泡到 UI 层。
- **设计理由**：合法的条件表达式深度通常不超过 10（如
  `(Can.baudrate == 500 && CanTrcv.type == "CAN_FD") || Can.baudrate == 250`）。50 的阈值留出了 5x 的安全余量，同时远低于 V8 栈深度限制。

### Layer 2: 循环依赖检测深度限制（MAX_DEPTH = 20）

**位置**：`conditions/depends.ts:DependencyGraph`

```typescript
static readonly MAX_DEPTH = 20;
```

- **触发条件**：DFS 遍历依赖图深度超过 20。
- **行为**：抛 Error（"MAX_DEPTH exceeded during cycle detection. Possible
  circular dependency near: {key}"）。
- **设计理由**：依赖图的拓扑深度通常不超过 3~5 层（A → B →
  C）。20 层的阈值足以覆盖所有现实场景，同时防止恶意输入在 `detectCycles()`
  中消耗大量 CPU。比条件表达式更保守（20 vs
  50），因为依赖图对性能更敏感且更容易被循环规则利用。

### Layer 3: Fail-Closed 策略（默认安全）

**位置**：`conditions/evaluator.ts:ConditionEvaluator`

对于无法解析的路径、缺失的模块配置、无效的类型比较，统一返回 `false`
而非抛出异常：

```typescript
// 未找到模块配置 → undefined → 布尔上下文中 false
const config = configs.find((c) => c.module === moduleName);
if (!config) return undefined; // → Boolean(undefined) → false

// 未知操作符 → false
default: return false;
```

这意味着当前模块的参数在条件无法求值的情况下会**隐藏**（fail-closed=隐藏/禁用），而不是显示（fail-open=显示/启用）。对于安全相关的 AutoSAR 配置场景，隐藏比误显示增加了用户的操作门槛，更安全（用户需要主动去掉条件才能看到参数）。

### 阈值对比

| 层级    | 组件               | MAX_DEPTH | 策略                           | 触发后果                       |
| ------- | ------------------ | --------- | ------------------------------ | ------------------------------ |
| Layer 1 | ConditionEvaluator | 50        | Error + 封装为 ValidationError | 单条表达式失败，不影响其他验证 |
| Layer 2 | DependencyGraph    | 20        | Error + 终止当前检测           | 依赖图无法构建，传播服务不可用 |
| Layer 3 | 全局路径解析       | N/A       | Fail-closed → false            | 参数隐藏/禁用，UI 不崩溃       |

## 后果

### 优点

1. **防御纵深**
   — 三层保护覆盖了从表达式求值到依赖图分析的完整路径，单点失效不会波及其他模块。
2. **合理的阈值预设** —
   20 和 50 的选择基于对 AutoSAR 配置真实场景的分析：没有合法表达式的深度超过 10，没有合法依赖链超过 5。阈值留有 2x~5x 余量。
3. **Fail-closed 优于 Fail-open**
   — 在无法判断参数可见性时默认隐藏，避免用户在错误的上下文中修改参数值。
4. **可追溯的错误报告**
   — 所有熔断事件都有清晰的错误消息和位置信息，便于用户定位问题表达式或规则。

### 缺点

1. **硬编码阈值** —
   MAX_DEPTH 是编译时常量，不能按需调整。对于极端复杂的项目（如整车级 ARXML 聚合），50 层可能不够。
2. **Fail-closed 的副作用**
   — 如果 ARXML 解析 bug 导致路径前缀错误，相关参数会莫名其妙隐藏，用户可能认为是 UI
   bug 而非配置问题。
3. **依赖图熔断导致传播不可用** — 一旦 DependencyGraph 抛出 MAX_DEPTH 异常，整个
   `ConstraintPropagator` 实例进入不可用状态，需要重建。

### 未来改进

- 将 MAX_DEPTH 改为可配置的选项（`evaluatorOptions.maxDepth`）。
- 为 DependencyGraph 增加熔断恢复机制（重置状态后重试）。
- 在 fail-closed 的 UI 控件上添加说明 tooltip（"此参数因条件求值失败而隐藏"）。

## 备选方案

### 方案 A: 无限制递归（已否决）

不设置深度限制，依赖 V8 原生栈溢出处理。

- **问题**：`RangeError: Maximum call stack size exceeded`
  是不可捕获的致命错误，会导致整个进程崩溃。在 Electron/CLI 工具中这是不可接受的。

### 方案 B: 用迭代代替递归（已否决）

将所有递归求值改为显式栈迭代，消除栈溢出风险。

- **问题**：AST 递归求值是表达力最强、最易维护的实现方式。改为迭代栈需要维护状态机，代码复杂度急剧上升且不利于类型推导。用深度限制（50 层）已经覆盖了所有合法场景，迭代带来的好处不值代码复杂度的代价。

### 方案 C: 单一大阈值（如 MAX_DEPTH = 2000，已否决）

对所有组件使用统一的 2000 深度限制。

- **问题**：依赖图检测的 worst-case 复杂度是指数级的，2000 层深度可能导致
  `detectCycles()` 运行数秒甚至数分钟。分层阈值（条件 50
  / 依赖图 20）是为了对依赖图施加更严格的控制。
