# ADR-003: 跨模块验证器架构

## 状态

已采纳 · 2025-08-15

## 背景

在 AutoSAR 配置系统中，单个模块的参数往往受到其他模块参数值的约束。例如：

- `Can.baudrate` 必须在 `CanTrcv.supported_baudrates` 范围内
- `Mcu.clockFrequency` 必须与 `Can.syncClock` 一致
- `Os.task.priority` 不能超过 `Os.resource.priorityCeiling`

yuleASR-Configurator 的 `ModuleParameter` 通过 `crossReferences` 字段声明这些约束。验证器需要在运行时检查它们是否满足。

核心设计问题有：

1. **如何构造验证器？** — 验证依赖于 `ModuleSchema`（从中读取 `crossReferences` 和参数约束），所以需要一个构建阶段将 Schema 注入。
2. **验证范围如何选择？** — 全量验证 50+ 模块的数千条约束代价高；而用户在 UI 中一次只修改一个参数，需要局部验证。

## 决策

### 1. 工厂函数模式而非构造函数注入

使用 `createCrossModuleValidator(schemas: ModuleSchema[])` 工厂函数替代直接 `new CrossModuleValidator(schemaMap)`。

```typescript
// 工厂函数
export function createCrossModuleValidator(
  schemas: ModuleSchema[]
): CrossModuleValidator {
  const schemaMap = new Map(schemas.map(s => [s.name, s]));
  return new CrossModuleValidator(schemaMap);
}
```

- **理由**：`ModuleSchema[]` 是更自然的输入（用户手头通常是数组而非 Map）；工厂函数封装了 `Array → Map` 的转换逻辑，调用方不需要知道内部实现。
- **对比直接 `new`**：调用方不需要知道构造参数类型是 `Map<string, ModuleSchema>`，降低了 API 的认知门槛。

### 2. 增量验证模式（validateAffectedBy）

除了全量 `validate()` 外，提供 `validateAffectedBy()` 方法，只验证与指定参数相关的约束。

```typescript
validateAffectedBy(
  changedParams: Array<{ module: string; param: string }>,
  configs: ModuleConfig[]
): ValidationError[]
```

- **实现策略**：
  - **(A) 正向查询**：被修改参数自身的 `crossReferences` 约束。
  - **(B) 反向查询**：其他模块中引用该参数的 `crossReferences` 约束（`ref.module === changedModule && ref.param === changedParam`）。
- **效率**：O(numChangedParams × numCrossRefs)，与实际修改的参数数量线性相关，与总模块数量无关。

### 3. 三阶段验证管道（ValidationPipeline）

在 `validators/validation-pipeline.ts` 中包装三个验证阶段：

```
ValidationPipeline.validate()
  ├── 1. YuleasrValidator.validateModules()  ← 模块级（必需参数、类型、范围）
  ├── 2. CrossModuleValidator.validate()     ← 跨模块引用约束
  └── 3. evaluateCondition() 捕获异常       ← 条件表达式语法/运行时错误
```

每个阶段输出独立的错误列表并在最终合并去重（按 `path|message` 去重）。

## 后果

### 优点

1. **局部可用、全局可用** — UI 修改单个参数时可调用 `validateAffectedBy` 获得毫秒级验证；提交配置时调用 `validate` 做全量确认。
2. **错误分类清晰** — 模块级错误、跨模块错误、条件错误分别归档，前端 UI 可以分别渲染在不同的校验面板中。
3. **工厂函数封装性好** — 调用方不需要理解 `Map` 的构建；如果未来需要切换 Schema 存储结构（如 WeakMap），只需修改工厂函数一个点。
4. **跨模块约束声明式** — 约束在 `ModuleParameter.crossReferences` 中声明而非硬编码在验证逻辑中，新增约束不需要修改验证器代码。

### 缺点

1. **仅支持五类关系** — `equals / less_than / greater_than / in_range / in_enum` 覆盖了 95% 的 AutoSAR 跨模块约束场景，但少数复杂约束（如位掩码对齐、校验和计算）需要自定义验证器插件。
2. **反向查询是线性扫描** — `validateAffectedBy` 的反向查询需要遍历所有其他模块的参数，在有 50+ 模块、每个模块 100+ 参数时可能成为瓶颈。未来可以引入反向索引优化。

## 备选方案

### 方案 A: 单一全量验证器（已否决）

不区分全量和增量，每次验证都扫描所有模块的所有参数。

- **问题**：用户在 UI 中每次按键都触发一次全量验证，50 模块 × 100 参数 × 5 条约束 = 25,000 次检查/操作，导致 UI 卡顿。

### 方案 B: 验证规则内嵌在验证器代码中（已否决）

将跨模块约束写成固定逻辑，而不是读取 schema 上的 `crossReferences`。

- **问题**：每引入一个新模块都需要修改验证器源码，违反开闭原则。
