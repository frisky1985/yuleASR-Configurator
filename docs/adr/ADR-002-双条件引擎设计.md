# ADR-002: 双条件引擎设计

## 状态

已采纳 · 2025-08-15

## 背景

yuleASR-Configurator 需要处理两种本质上不同的参数间关系：

1. **可见性/启用性条件（UI 语义）**
   — 参数 A 的值控制参数 B 是否在表单中显示或可编辑（`visibleWhen` /
   `enabledWhen`）。这类条件的结果是布尔值，仅在 UI 层消费。
2. **值传播约束（配置语义）** — 参数 A 的值变化后，参数 B 的
   `min`/`max`/`default`/`options`
   必须动态更新（例如 Can.baudrate 变化后，CanTrcv 的 supported_baudrates 选项需要过滤）。这类传播不是布尔判定，而是数据变换。

最初的设计考虑过将两者合并到一个"通用条件引擎"中，但在评估中发现两者的求值语义、数据流方向、失败处理策略有根本差异。

## 决策

将条件引擎拆分为两个独立的子系统，各自专精一种场景：

### ConditionEvaluator（`conditions/evaluator.ts`）

- **职能**：解析并求值条件表达式 DSL AST，产生布尔结果。
- **输入**：条件表达式字符串（如 `"Can.baudrate == 500"`），经
  `parseCondition()` 解析为 AST。
- **输出**：`boolean`（true/false），fails-closed（无法解析的路径返回 false）。
- **消费方**：UI 渲染层，决定控件的可见/禁用状态。
- **核心特性**：AST 递归求值（`evaluateNode`）、短路操作符（`||`、`&&`）、三段路径解析（`module.container[index].param`）。

### ConstraintPropagator（`conditions/propagator.ts`）

- **职能**：根据 PropagationRule 定义，将源参数的值变化传播到目标参数的约束字段上。
- **输入**：`ChangedParams[]`（变化的参数列表）+
  `PropagationRule[]`（传播规则）。
- **输出**：`PropagationResult[]`（描述每个字段的旧值和新值）。
- **消费方**：应用核心层，在用户修改参数后同步更新其他模块的 Schema。
- **核心特性**：`DependencyGraph（conditions/depends.ts）`拓扑排序、循环依赖检测与警告、`{source}`
  占位符表达式求值（支持四则运算、三元表达式、比较）。

### 分离依据

| 维度       | ConditionEvaluator          | ConstraintPropagator            |
| ---------- | --------------------------- | ------------------------------- |
| 求值结果   | 布尔值                      | 数值/数组/任意值                |
| 副作用     | 无（纯函数）                | 有（修改 Schema 字段）          |
| 数据流     | 读入方向（从 config 到 UI） | 写入方向（从 config 到 schema） |
| 失败策略   | fail-closed（false）        | 运行时异常 + 循环检测警告       |
| 表达式语法 | 完整的条件 DSL              | 数学表达式 + `{source}` 替换    |
| 执行时机   | 每次 UI 渲染                | 参数值变更后                    |

## 后果

### 优点

1. **关注点分离** — 两个引擎可以各自独立优化和测试，没有条件分支互相干扰。
2. **错误隔离** —
   UI 条件求值的 fail-closed 策略不会影响配置数据的完整性；值传播的失败（如循环依赖）不会破坏 UI 渲染路径。
3. **类型安全** — `PropagationTarget.field` 被限定为
   `'min' | 'max' | 'default' | 'options'`
   四个字面量，TypeScript 编译器可保证传播目标字段的合法性。
4. **可测试性** — 纯函数求值的 `ConditionEvaluator` 和带副作用的
   `ConstraintPropagator` 可以分别用不同的测试策略覆盖。

### 缺点

1. **概念复杂度** — 新开发者需要理解两个引擎的存在和各自的使用场景。
2. **表达式 DSL 不统一**
   — 条件表达式和传播表达式有不同的语法，用户需要学习两套 DSL。

### 缓解措施

- 在 API 文档中明确区分两类条件的用途。
- 前端 UI 中通过不同的编辑器控件（条件用代码编辑器 + 语法高亮，传播用可视化的规则配置表单）降低认知负担。

## 备选方案

### 方案 A: 统一条件引擎（已否决）

用一个通用的 `ConditionEngine` 处理所有参数间关系，通过 `mode`
参数区分是返回布尔值还是执行传播。

- **问题**：单一引擎需要同时处理布尔求值、数据变换、收拢/发散两种数据流方向，导致核心逻辑被条件分支污染，违反单一职责原则。

### 方案 B: 仅保留 ConstraintPropagator，UI 条件用传播模拟（已否决）

将 `visibleWhen` 也建模为一个传播规则，传播到 `ui.hidden` 字段上。

- **问题**：
  - 传播是 O(n) 全量扫描匹配规则，而 UI 条件只需要 O(1) 路径查询。
  - 传播修改 Schema 字段会触发重渲染循环，违反 React 的数据流原则。
  - 无法利用条件 DSL 的短路优化。
