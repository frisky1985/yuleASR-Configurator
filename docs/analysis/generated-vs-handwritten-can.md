# Gap Analysis: ecuc-generator vs yuleASR Can_Cfg.h

## 结论

**ecuc-generator 和 yuleASR 的 Can_Cfg.h 是互补的，不是冲突的。**

- **yuleASR 已有的 `Can_Cfg.h`** — 是一个 **配置常亮头文件**（纯
  `#define`），给已经写好的 Can 驱动提供参数
- **ecuc-generator 的输出** — 是一个
  **完整 AUTOSAR 模块代码包**，包括驱动接口声明、数据结构、初始化/反初始化函数

两个东西解决不同的层次，对接后正好互相补充。

---

## 逐项对比

| 维度         |      yuleASR Can_Cfg.h       |              ecuc-generator 输出               | 结论               |
| ------------ | :--------------------------: | :--------------------------------------------: | :----------------- |
| **文件角色** |         驱动配置参数         |             完整模块代码（头+源）              | 互补               |
| **Guard**    |         `CAN_CFG_H`          |                  `ECUC_CAN_H`                  | ⚠️ 需对齐          |
| **文件名**   |         `Can_Cfg.h`          |                  `Ecuc_Can.h`                  | ⚠️ 需对齐          |
| **包含**     |          无 include          |            `Std_Types.h`, `Ecuc.h`             | 互补               |
| **宏**       | 配置常数（波特率、数量、ID） |               Module ID + 版本宏               | 互补               |
| **类型定义** |              无              |   `Can_ConfigType`, `Can_CanControllerType`    | 互补               |
| **函数声明** |              无              | `Can_Init`, `Can_DeInit`, `Can_GetVersionInfo` | 互补               |
| **注释**     |        YuleTech 版权         |               AUTOGENERATED 标记               | 需整合             |
| **文件数量** |             1 个             |          4 个（.h/.c/\_PBcfg/\_Lcfg）          | 多出的文件是新能力 |

## 对接方案

ecuc-generator 应：

1. 输出 `Can_Cfg.h`（用目标文件名，不是 `Ecuc_Can.h`）
2. 使用 `CAN_CFG_H` guard（匹配现有项目）
3. **只生成现有 Can_Cfg.h 不包含的内容**（类型+函数），宏部分从 Web 层 codegen 补充

这意味着 ecuc-generator 实际上是在 **生成现有的
`Can_Cfg.h`（宏部分）+ 新增的驱动代码骨架**。

## 数据流

```
Configurator UI
     │
     ▼
Web codegen (codegen.ts) ──→ Can_Cfg.h (#define constants)
     │
     ▼
ecuc-generator ──→ Can.c + Can.h + Can_PBcfg.c + Can_Lcfg.c (types + functions)
     │
     ▼
yuleASR CMake build ──→ 编译成功
```
